关于C#中常用的数据结构性能比较

Red Coding
2019-09-24 (Updated: 2019-12-04)
U3D
C#

常用的数据结构

我们开发中用的最多的数据结构无非就是:Arry、List、Stack、Queue以及Dictionary，那么我们什么时候用List什么时候用Stack呢？可能很多工作了几年的老手都弄不明白，只是知道这个能满足我的业务需求就用这个，性能、优化都会抛之脑后，写出的代码性能低下难以优化，最后不得不重构。接下来我们从源码角度来剖析一下它们的实现并分析一下什么情况下用什么类型的数据结构，争取以后写出的代码摆脱重构的“命运”…

List、Stack、Queue横向对比

看过List、Stack、Queue源码后我们知道其内部都是用数组来实现的，所以我们放在一起来做一下横向对比。

初始化

从上面的源码中可以看出，当我们使用new关键词初始化这三种类型的数据结构时,其实是在内部初始化了一个数组。只不过Stack和Queue是Object类型的数组而List是泛型数组。那么接下来我们看一下初始化的默认大小是多少？

可以看出Stack给出的是10（之前net版本是4）、Queue一开始没有给出默认值在Enqueue时动态计算，通过看源码得出默认容量是4、List的给出的也是4，也就是说使用List或Queue进行添加操作时List和Queue会比Stack优先扩容。

添加

我们都知道Stack添加数据的方法是Push,Queue的方法是Enqueue,List是Add,接下来我们看看它们的内部实现：

我们可以看出以上三种数据结构进行添加操作时都是检查数组容量够不够用，不够用则对数组进行扩容，够用则直接增加索引值，不同的是Stack和Queue是new一个新的数组用数组拷贝的方法进行扩容，而List是计算索引对原数组进行扩容，这一点上List优于前两者，减少了内存开销为GC减轻了负担。
接下来我们继续看是怎么进行扩容的，Stack的默认数组大小是10而每次扩容都是数组长度的两倍，扩容路线也就是：10，20，40，80，160…而Queue默认的数组大小是4每次扩容 (int)((long)_array.Length * (long)_growFactor / 100)也就是(4 * 200 /100) = 8，可以得出Queue的扩容也是数组长度的两倍，扩容路线是：4，8，16…List的扩容和Queue一样都是原来的2倍。

获取元素

从源码可以看出三者都是直接从数组里面获取索引，只不过Stack是从尾部获取，Queue是从头部开始，而Stack和Queue除了Peek()方法获取元素外还可以用Pop()和Dequeue()方法获取，获取的原理和Peek()相同只不过获取的同时从数组里面移除了该元素。

包含

包含操作就是遍历数组对每个元素进行比较，相等返回true不相等返回false,这里要注意的是Stack和Queue都是用的while循环都是从尾部开始匹配的，而List用的是for循环是从头部开始匹配的。

移除

Stack和Queue上面已经说过了，这里只说一下List的移除操作

Remove接口中包含了 IndexOf 和 RemoveAt，用 IndexOf 找到元素的索引位置，用 RemoveAt 删除指定位置的元素。删除的原理其实就是用 Array.Copy 对数组进行覆盖。IndexOf 启用的是 Array.IndexOf 接口来查找元素的索引位置，这个接口内部实现是就是按索引顺序从0到n对每个位置的比较，复杂度为O(n)。

Clear

三者的方式基本相同Clear时并不会删除数组，而是将数组中的元素清空，并置 _size 为 0，表明当前容量为0。

ToArry

ToArry操作都是通过new一个新的数组，然后通过数组拷贝最后把值返回出来的。

总结

  • 由以上源码可以看出,在性能上 Stack > Queue > List，根据需求我们应当优选Stack,然后在考虑Queue最后考虑List
  • Stack、Queue、List都是用数组形式作为底层数据结构，好处是使用索引方式提取元素很快，但在扩容的时候就很糟糕了，每次new数组都会造成内存垃圾，这给垃圾回收GC带来了很多负担。一次性扩充更多的数组容量为GC减轻了负担，但数组连续的new也还是GC的不小负担，特别是代码中List频繁使用的Add时。另外，如果数量不得当也会浪费大量内存空间，比如当元素数量为 520 时，数组空间就会扩容到1024个元素，如果不使用剩余的504个空间单位，就造成了大部分的内存空间的浪费。所以初始化的时候尽量给足所需空间，避免自动扩容时造成内存浪费。
